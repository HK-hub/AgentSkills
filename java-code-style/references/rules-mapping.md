# 阿里规范映射表（v0.2）

将《阿里巴巴Java开发规范（嵩山版）》映射到 `java-code-style` 的执行等级。

## 分级映射规则

- 手册【强制】→ `L1 Blocking`（必须整改）
- 手册【推荐】→ `L2 Important`（建议整改）
- 手册【参考】→ `L3 Nice-to-have`（可选优化）

## A. 命名与常量

### L1 Blocking

1. 命名禁止以下划线/美元符号开头或结尾。
   参考：第4页，命名风格-1
2. 禁止中英文/拼音混用命名，禁止中文标识符。
   参考：第4页，命名风格-2
3. 类名 UpperCamelCase，方法/变量 lowerCamelCase。
   参考：第4页，命名风格-4/5
4. 常量全大写+下划线分隔。
   参考：第4页，命名风格-6
5. 包名全小写，语义单词分段。
   参考：第5页，命名风格-10
6. 禁止魔法值直接出现在代码中。
   参考：第7页，常量定义-1

### L2 Important

1. 使用可解释的完整单词组合，避免生硬缩写。
   参考：第5-6页，命名风格-12/13
2. 常量按功能域分类维护，避免“巨型常量类”。
   参考：第7页，常量定义-3

### L3 Nice-to-have

1. 枚举名建议 `Enum` 后缀，成员全大写。
   参考：第6页，命名风格-18
2. Service/DAO 命名采用 get/list/count/save/update/remove 语义前缀。
   参考：第6页，命名风格-19

## B. 代码格式与可读性

### L1 Blocking

1. if/for/while/switch/do 必须使用大括号。
   参考：第24页，控制语句-3
2. 统一4空格缩进，禁止Tab。
   参考：第8页，代码格式-5
3. switch 必须包含 default；case 必须显式终止或注释贯穿。
   参考：第23-24页，控制语句-1

### L2 Important

1. 异常分支优先卫语句，控制嵌套层级（建议不超3层）。
   参考：第25页，控制语句-7
2. 不同语义代码块间用单空行分隔，避免多空行。
   参考：第10页，代码格式-13

## C. OOP 与对象设计

### L1 Blocking

1. 覆写方法必须 `@Override`。
   参考：第10页，OOP规约-2
2. 外部依赖接口禁止直接改方法签名；废弃接口需 `@Deprecated` 且给替代方案。
   参考：第10页，OOP规约-4
3. equals 调用采用常量/确定非空对象，避免 NPE。
   参考：第10页，OOP规约-6
4. BigDecimal 禁止 `new BigDecimal(double)`，等值比较用 `compareTo()`。
   参考：第12页，OOP规约-10/12
5. POJO 不设置属性默认值；禁止同属性同时出现 `isXxx()` 和 `getXxx()`。
   参考：第12页，OOP规约-14/18
6. 构造方法禁止业务逻辑。
   参考：第12页，OOP规约-16

### L2 Important

1. 多个同名方法/构造器集中放置，方法顺序清晰。
   参考：第13页，OOP规约-20/21
2. setter/getter 中不放业务逻辑。
   参考：第13页，OOP规约-22

## D. 集合、并发与控制流

### L1 Blocking

1. 自定义对象覆写 equals 时必须覆写 hashCode。
   参考：第15页，集合处理-1
2. 线程资源必须来自线程池；禁止显式创建线程。
   参考：第20页，并发处理-3
3. 禁止 `Executors` 直接建线程池，使用 `ThreadPoolExecutor` 明确参数。
   参考：第20页，并发处理-4
4. ThreadLocal 在线程池场景必须回收（try-finally remove）。
   参考：第21页，并发处理-6
5. 加锁与解锁结构必须严格成对，避免异常路径泄漏锁。
   参考：第21-22页，并发处理-9/10
6. 并发更新同记录必须防更新丢失（应用锁/缓存锁/DB乐观锁）。
   参考：第22页，并发处理-11

### L2 Important

1. 线程命名可回溯；按业务特征分组。
   参考：第20页，并发处理-2
2. 同步范围最小化，优先无锁/细粒度锁。
   参考：第21页，并发处理-7

## E. 异常、错误码、日志

### L1 Blocking

1. 不应通过 catch 处理可预检查规避的 RuntimeException。
   参考：第33页，异常处理-1
2. 禁止用异常做流程控制。
   参考：第33页，异常处理-2
3. 捕获异常必须处理；不处理则上抛，最外层转用户可理解信息。
   参考：第33页，异常处理-4
4. finally 中必须关闭资源；禁止在 finally 中 return。
   参考：第34页，异常处理-6/7
5. 日志必须使用门面（SLF4J/JCL），禁止直接依赖实现API。
   参考：第35页，日志规约-1
6. 生产环境禁止 `System.out`/`System.err`/`printStackTrace`。
   参考：第36页，日志规约-8
7. 日志参数使用占位符，避免字符串拼接。
   参考：第36页，日志规约-5
8. 异常日志需包含现场信息 + 异常堆栈。
   参考：第36页，日志规约-9
9. 错误码不能直接展示给用户。
   参考：第32页，错误码-7

### L2 Important

1. 避免 `new RuntimeException()` 泛化抛错，定义业务语义异常。
   参考：第35页，异常处理-12
2. 日志分级审慎：debug谨慎，info有选择，warn避免泛滥。
   参考：第36-37页，日志规约-11/12

## F. 安全、SQL、ORM

### L1 Blocking

1. 用户输入参数必须做有效性校验。
   参考：第40页，安全规约-4
2. SQL 参数必须绑定，禁止字符串拼接SQL。
   参考：第40页，安全规约-3
3. 禁止输出未过滤/未转义的用户输入到HTML。
   参考：第40页，安全规约-5
4. 表单/AJAX 必须做 CSRF 校验。
   参考：第40页，安全规约-6
5. 查询字段禁止 `SELECT *`。
   参考：第45页，ORM映射-1
6. MyBatis/iBatis 参数使用 `#{}`，禁止 `${}`。
   参考：第46页，ORM映射-4
7. update 时必须同步更新 `update_time`。
   参考：第46页，ORM映射-7
8. SQL 订正先 select 后 update/delete。
   参考：第44页，SQL语句-8

### L2 Important

1. 避免 in 超大集合，建议控制在1000以内。
   参考：第45页，SQL语句-11
2. 建组合索引时优先高区分度列，兼顾等值条件前置。
   参考：第43页，索引规约-9

## G. 工程分层与设计

### L1 Blocking

1. 核心存储方案与底层数据结构须评审并文档化。
   参考：第51页，设计规约-1

### L2 Important

1. 分层职责清晰：Web 参数校验/转发，Service 业务逻辑，DAO 数据访问。
   参考：第47页，应用分层-1
2. 设计优先单一职责、依赖倒置、开闭原则。
   参考：第52页，设计规约-9/11/12

  2) 团队约定
  3) 手册规则
  4) 个人偏好
